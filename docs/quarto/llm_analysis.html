Overview

This page describes the large language model (LLM) workflow used to extract structured information from PDFs of studies related to the United States Renal Data System (USRDS). The goal was to evaluate whether LLMs could reproduce the same kinds of abstractions performed during manual review, including study classification, programming language, files used, funding, and other study characteristics.

Each PDF was processed in R using the ellmer package. The text of each paper was extracted from the PDF, submitted to an LLM together with a standardized prompt and schema, and returned in a structured format suitable for downstream analysis. This allowed the outputs of multiple models to be compared directly with one another and with human review.

Because the corpus was moderately large and API-based extraction can be interrupted by rate limits, transient failures, or incomplete runs, each workflow was built to support checkpointing, retry logic, and resumable processing. Intermediate results and job-status files were saved throughout so that processing could continue without restarting from the beginning.

The sections below show the scripts used for the three LLM pipelines evaluated in this project: ChatGPT, Claude, and Google Gemini.

For details on how to configure ellmer to work with LLMs via the APIs, see the package documentation.

Abstraction schema

To make LLM-based review reproducible and comparable across models, we defined a common abstraction schema in R and used it for all three LLM pipelines. In practice, this schema had two linked components. The first was the instruction text sent to the model with each paper, which defined the extraction task, imposed decision rules, and limited when the model was allowed to mark a field as present. The second was the typed output schema used by ellmer, which specified the exact variables to be returned and the data type of each variable. Together, these components transformed the task from open-ended summarization into structured data abstraction.

The instruction text was designed to reduce hallucination and improve consistency across papers. Models were told that they were acting as expert reviewers of USRDS-based research papers and were instructed to extract metadata using strict, sequential, and anti-hallucination rules. They were told to return only the fields defined in the schema, with no commentary and no reasoning. They were also explicitly told not to infer key details that are often omitted or ambiguously reported in manuscripts, including programming language, crosswalk use, and specific USRDS files used. When such details were not explicitly stated, the instructions required the model to return FALSE, NULL, or an allowed uncertainty category rather than guessing.

The prompt text also encoded decision logic that guided the order and interpretation of extraction. It instructed the model to proceed sequentially: first extract bibliographic metadata, then determine whether the paper represented a new data analysis, then determine whether the study was transplant-related, and then classify more detailed variables such as study subtype, programming languages, files used, and component-level data elements. This sequencing was intended to mimic the logic of human review and reduce inconsistent downstream classification. For example, the prompt specified that a paper should be classified as a new data analysis only if it conducted original analyses using USRDS or Medicare ESRD patient-level data. It also specified that transplant-related status should be marked TRUE only when transplantation itself was explicitly studied, not when transplant date was mentioned only as a censoring event.

Several prompt rules were designed as guardrails for downstream fields. If the text did not explicitly mention USRDS or Medicare ESRD program data, the model was instructed to set the new-analysis flag to FALSE, set all file-use and component variables to FALSE, mark file appropriateness as FALSE, mark crosswalk use as FALSE, and return no unsupported-task list for the package coverage field. In other words, the prompt contained not only extraction instructions but also internal consistency rules intended to keep related fields aligned with one another. This was important because many papers mention kidney disease populations or administrative data in ways that could tempt a model to overgeneralize from context.

The prompt also contained explicit definitions for important variable groups. For programming language, the model was told to mark a language as present only if it was directly named in the paper, and not to infer language from generic phrases such as “statistical software” or mentions of common packages. For file identification, the prompt gave operational definitions for several USRDS file domains, including Core, Transplant, Part D, Institution, Physician/Supplier, Hospital, and CROWNWeb. For example, Core was linked to concepts such as the 2728 form, ESRD onset, baseline demographics, residence, payer, and transplant date, whereas Transplant was linked to donor type, graft failure, and waitlist information. Similarly, the prompt defined component-level variables such as basic demographics, ICD-9, ICD-10, CPT codes, Part D medications, transplant-specific information, costs, and residential information. These definitions were intended to make model classification more consistent and more closely aligned with how a knowledgeable human reviewer would interpret the paper.

A particularly important feature of the prompt was its distinction between explicit reporting and inferred file source. The model was required to classify file source type as explicit when USRDS was directly named, inferred when Medicare ESRD claims use was implied but USRDS was not directly named, and uncertain when the source could not be determined confidently. This allowed the pipeline to capture a meaningful distinction between papers that clearly documented their use of USRDS and papers in which the likely data source had to be inferred from context. That distinction would have been difficult to preserve in a free-text summary and was therefore encoded directly into the schema.

The second half of the schema was the typed ellmer object, defined as type_summary. This object formally declared the expected output fields and their data types. Bibliographic fields such as authors, title, and journal were represented as strings, while publication year was represented as an integer. Many of the review variables were encoded as booleans, including whether the paper was a new data analysis, whether it was transplant-related, whether the authors stated a programming language, whether specific files were used, whether specific data components were abstracted, whether funding was reported, whether code was shared, and whether crosswalks were used. Fields with a limited set of allowed values were encoded as enumerations. For example, transplant study type was restricted to four values: Pre-transplant access to care, Pre-transplant management/outcomes, Post-transplant, or NA. File source type was likewise restricted to explicit, inferred, or uncertain.

This typed schema mattered methodologically because it constrained model output into a form that could be analyzed directly in R. Rather than parsing narrative text after the fact, the pipeline requested structured outputs that matched the predefined variable set. This made the outputs from ChatGPT, Claude, and Gemini directly comparable with one another and with human review. It also reduced ambiguity in downstream analysis, since the model could not invent new field names or return arbitrary prose in place of a boolean or enumerated response. In effect, the typed schema served as both a validation layer and a bridge between model output and statistical analysis.

The variables included in type_summary closely mirrored the goals of the manual abstraction process, but also extended it in a few analytically useful ways. In addition to bibliographic metadata and study classification, the typed schema included a field for the type of transplant study, a three-level file-source variable, individual booleans for each programming language, booleans for each major USRDS file domain, booleans for each component-level data type, a short text field giving the reason why file choice did or did not seem appropriate, a free-text comments field, a funding field, package-coverage variables, and separate booleans for individual crosswalk types. This level of granularity allowed the LLM review to function not only as a test of model performance, but also as a structured characterization of how USRDS is used in the literature.

The schema also linked directly to the broader aims of the project. Because one study goal was to understand what kinds of analytical tasks might be supported by the proposed usRds package, the prompt and typed schema included fields asking whether the major data tasks in a given paper would be covered by the package and, if not, which tasks would fall outside its scope. Likewise, because papers sometimes rely on linkage resources that are not obvious from the main text, the schema included dedicated fields for crosswalk use, including separate booleans for UNOS, physician, and provider crosswalks. These variables allowed the LLM pipeline to capture not just what the paper studied, but also what kinds of data engineering and linkage work were required to perform the study.

Taken together, the prompt text and the ellmer typed schema formed a single methodological framework. The prompt defined the rules of extraction, including what counted as sufficient evidence and how ambiguous cases should be handled, while the typed schema defined the exact structure of the final result. Using both components together helped standardize model behavior, minimize overinterpretation, and produce outputs that could be summarized quantitatively and compared directly with manual abstraction.

extract_summary_row <- function(pdf_path,
                                initial_max_chars = 240000,
                                min_chars = 5000,
                                initial_wait = 300,
                                max_wait = 1200,
                                max_retries = 5) {
  if (!file.exists(pdf_path)) {
    message("Skipping missing file: ", pdf_path)
    return(tibble(filename = basename(pdf_path)))
  }
  
  text_pages <- pdftools::pdf_text(pdf_path)
  full_text <- paste(text_pages, collapse = "\n")
  full_char_length <- nchar(full_text)
  max_chars <- min(full_char_length, initial_max_chars)
  wait_time <- initial_wait
  retries <- 0
  
  while (max_chars >= min_chars && retries <= max_retries) {
    input_text <- substr(full_text, 1, max_chars)
    input_token_est <- estimate_tokens(input_text)
    message(Sys.time(), " 🔄 Attempting ", basename(pdf_path), " with ~", input_token_est, " tokens")
    
    result <- tryCatch({
      chatgpt1 <- chat_openai(
        model = "gpt-5.1",
        system_prompt = id_prompt("")
      )
      chatgpt1$chat_structured(input_text, type = type_summary)
    }, error = function(e) {
      if (grepl("tokens per min|quota|429", e$message)) {
        message("⚠️ Rate limit or quota error. Waiting ", wait_time, "s")
        total_retries <<- total_retries + 1
        Sys.sleep(wait_time)
        return("RATE_LIMIT_RETRY")
      } else {
        message("❌ Error: ", e$message)
        return(structure("ERROR", class = "error"))
      }
    })
    
    if (identical(result, "RATE_LIMIT_RETRY")) {
      retries <- retries + 1
      wait_time <- min(wait_time * 1.5, max_wait)
      max_chars <- floor(max_chars * 0.9)
      next
    }
    
    if (inherits(result, "error") || is.null(result)) {
      chatgpt1 <- chat_openai(
        model = "gpt-5.1",
        system_prompt = id_prompt("")
      )
      fallback <- chatgpt1$chat(input_text)
      output_token_est <- estimate_tokens(fallback)
      total_output_tokens <<- total_output_tokens + output_token_est
      message("⚠️ Fallback preview:\n", substr(fallback, 1, 500))
      break
    }
    
    output_token_est <- estimate_tokens(paste(result$title %||% "", result$journal %||% "", result$authors %||% ""))
    total_input_tokens <<- total_input_tokens + input_token_est
    total_output_tokens <<- total_output_tokens + output_token_est
    total_requests <<- total_requests + 1
    
    message("✅ Success: ", basename(pdf_path))
    return(as_tibble_row(
      c(
        list(filename = basename(pdf_path), text = input_text, char_length = nchar(input_text), raw_json = list(result)),
        map(result, ~ if (is.null(.x)) NA_character_ else paste(as.character(unlist(.x)), collapse = "; "))
      )
    ))
  }
  
  message("❌ Final failure for ", basename(pdf_path), " after ", retries, " retries.")
  tibble(filename = basename(pdf_path), text = substr(full_text, 1, min_chars), char_length = min_chars)
}

## Overview

This page describes the large language model (LLM) workflow used to extract structured information from PDFs of studies related to the United States Renal Data System (USRDS). The goal was to evaluate whether LLMs could reproduce the same kinds of abstractions performed during manual review, including study classification, programming language, files used, funding, and other study characteristics.

Each PDF was processed in R using the `ellmer` package. The text of each paper was extracted from the PDF, submitted to an LLM together with a standardized prompt and schema, and returned in a structured format suitable for downstream analysis. This allowed the outputs of multiple models to be compared directly with one another and with human review.

Because the corpus was moderately large and API-based extraction can be interrupted by rate limits, transient failures, or incomplete runs, each workflow was built to support checkpointing, retry logic, and resumable processing. Intermediate results and job-status files were saved throughout so that processing could continue without restarting from the beginning.

The sections below show the scripts used for the three LLM pipelines evaluated in this project: ChatGPT, Claude, and Google Gemini.

For details on how to configure `ellmer` to work with LLMs via the APIs, see the package documentation.

## Abstraction schema

To make LLM-based review reproducible and comparable across models, we defined a common abstraction schema in R and used it for all three LLM pipelines. In practice, this schema had two linked components. The first was the **instruction text sent to the model** with each paper, which defined the extraction task, imposed decision rules, and limited when the model was allowed to mark a field as present. The second was the **typed output schema used by `ellmer`**, which specified the exact variables to be returned and the data type of each variable. Together, these components transformed the task from open-ended summarization into structured data abstraction.

The instruction text was designed to reduce hallucination and improve consistency across papers. Models were told that they were acting as expert reviewers of USRDS-based research papers and were instructed to extract metadata using **strict, sequential, and anti-hallucination rules**. They were told to return only the fields defined in the schema, with no commentary and no reasoning. They were also explicitly told not to infer key details that are often omitted or ambiguously reported in manuscripts, including programming language, crosswalk use, and specific USRDS files used. When such details were not explicitly stated, the instructions required the model to return `FALSE`, `NULL`, or an allowed uncertainty category rather than guessing.

The prompt text also encoded **decision logic** that guided the order and interpretation of extraction. It instructed the model to proceed sequentially: first extract bibliographic metadata, then determine whether the paper represented a new data analysis, then determine whether the study was transplant-related, and then classify more detailed variables such as study subtype, programming languages, files used, and component-level data elements. This sequencing was intended to mimic the logic of human review and reduce inconsistent downstream classification. For example, the prompt specified that a paper should be classified as a new data analysis only if it conducted original analyses using USRDS or Medicare ESRD patient-level data. It also specified that transplant-related status should be marked `TRUE` only when transplantation itself was explicitly studied, not when transplant date was mentioned only as a censoring event. 

Several prompt rules were designed as **guardrails** for downstream fields. If the text did not explicitly mention USRDS or Medicare ESRD program data, the model was instructed to set the new-analysis flag to `FALSE`, set all file-use and component variables to `FALSE`, mark file appropriateness as `FALSE`, mark crosswalk use as `FALSE`, and return no unsupported-task list for the package coverage field. In other words, the prompt contained not only extraction instructions but also internal consistency rules intended to keep related fields aligned with one another. This was important because many papers mention kidney disease populations or administrative data in ways that could tempt a model to overgeneralize from context.

The prompt also contained explicit definitions for important variable groups. For programming language, the model was told to mark a language as present only if it was directly named in the paper, and not to infer language from generic phrases such as “statistical software” or mentions of common packages. For file identification, the prompt gave operational definitions for several USRDS file domains, including Core, Transplant, Part D, Institution, Physician/Supplier, Hospital, and CROWNWeb. For example, Core was linked to concepts such as the 2728 form, ESRD onset, baseline demographics, residence, payer, and transplant date, whereas Transplant was linked to donor type, graft failure, and waitlist information. Similarly, the prompt defined component-level variables such as basic demographics, ICD-9, ICD-10, CPT codes, Part D medications, transplant-specific information, costs, and residential information. These definitions were intended to make model classification more consistent and more closely aligned with how a knowledgeable human reviewer would interpret the paper. 

A particularly important feature of the prompt was its distinction between **explicit reporting and inferred file source**. The model was required to classify file source type as `explicit` when USRDS was directly named, `inferred` when Medicare ESRD claims use was implied but USRDS was not directly named, and `uncertain` when the source could not be determined confidently. This allowed the pipeline to capture a meaningful distinction between papers that clearly documented their use of USRDS and papers in which the likely data source had to be inferred from context. That distinction would have been difficult to preserve in a free-text summary and was therefore encoded directly into the schema.

The second half of the schema was the **typed `ellmer` object**, defined as `type_summary`. This object formally declared the expected output fields and their data types. Bibliographic fields such as authors, title, and journal were represented as strings, while publication year was represented as an integer. Many of the review variables were encoded as booleans, including whether the paper was a new data analysis, whether it was transplant-related, whether the authors stated a programming language, whether specific files were used, whether specific data components were abstracted, whether funding was reported, whether code was shared, and whether crosswalks were used. Fields with a limited set of allowed values were encoded as enumerations. For example, transplant study type was restricted to four values: `Pre-transplant access to care`, `Pre-transplant management/outcomes`, `Post-transplant`, or `NA`. File source type was likewise restricted to `explicit`, `inferred`, or `uncertain`.

This typed schema mattered methodologically because it constrained model output into a form that could be analyzed directly in R. Rather than parsing narrative text after the fact, the pipeline requested structured outputs that matched the predefined variable set. This made the outputs from ChatGPT, Claude, and Gemini directly comparable with one another and with human review. It also reduced ambiguity in downstream analysis, since the model could not invent new field names or return arbitrary prose in place of a boolean or enumerated response. In effect, the typed schema served as both a validation layer and a bridge between model output and statistical analysis. 

The variables included in `type_summary` closely mirrored the goals of the manual abstraction process, but also extended it in a few analytically useful ways. In addition to bibliographic metadata and study classification, the typed schema included a field for the type of transplant study, a three-level file-source variable, individual booleans for each programming language, booleans for each major USRDS file domain, booleans for each component-level data type, a short text field giving the reason why file choice did or did not seem appropriate, a free-text comments field, a funding field, package-coverage variables, and separate booleans for individual crosswalk types. This level of granularity allowed the LLM review to function not only as a test of model performance, but also as a structured characterization of how USRDS is used in the literature.

The schema also linked directly to the broader aims of the project. Because one study goal was to understand what kinds of analytical tasks might be supported by the proposed `usRds` package, the prompt and typed schema included fields asking whether the major data tasks in a given paper would be covered by the package and, if not, which tasks would fall outside its scope. Likewise, because papers sometimes rely on linkage resources that are not obvious from the main text, the schema included dedicated fields for crosswalk use, including separate booleans for UNOS, physician, and provider crosswalks. These variables allowed the LLM pipeline to capture not just what the paper studied, but also what kinds of data engineering and linkage work were required to perform the study. 

Taken together, the prompt text and the `ellmer` typed schema formed a single methodological framework. The prompt defined the rules of extraction, including what counted as sufficient evidence and how ambiguous cases should be handled, while the typed schema defined the exact structure of the final result. Using both components together helped standardize model behavior, minimize overinterpretation, and produce outputs that could be summarized quantitatively and compared directly with manual abstraction.

extract_summary_row <- function(pdf_path,
                                initial_max_chars = 240000,
                                min_chars = 5000,
                                initial_wait = 300,
                                max_wait = 1200,
                                max_retries = 5) {
  if (!file.exists(pdf_path)) {
    message("Skipping missing file: ", pdf_path)
    return(tibble(filename = basename(pdf_path)))
  }
  
  text_pages <- pdftools::pdf_text(pdf_path)
  full_text <- paste(text_pages, collapse = "\n")
  full_char_length <- nchar(full_text)
  max_chars <- min(full_char_length, initial_max_chars)
  wait_time <- initial_wait
  retries <- 0
  
  while (max_chars >= min_chars && retries <= max_retries) {
    input_text <- substr(full_text, 1, max_chars)
    input_token_est <- estimate_tokens(input_text)
    message(Sys.time(), " 🔄 Attempting ", basename(pdf_path), " with ~", input_token_est, " tokens")
    
    result <- tryCatch({
      chatgpt1 <- chat_openai(
        model = "gpt-5.1",
        system_prompt = id_prompt("")
      )
      chatgpt1$chat_structured(input_text, type = type_summary)
    }, error = function(e) {
      if (grepl("tokens per min|quota|429", e$message)) {
        message("⚠️ Rate limit or quota error. Waiting ", wait_time, "s")
        total_retries <<- total_retries + 1
        Sys.sleep(wait_time)
        return("RATE_LIMIT_RETRY")
      } else {
        message("❌ Error: ", e$message)
        return(structure("ERROR", class = "error"))
      }
    })
    
    if (identical(result, "RATE_LIMIT_RETRY")) {
      retries <- retries + 1
      wait_time <- min(wait_time * 1.5, max_wait)
      max_chars <- floor(max_chars * 0.9)
      next
    }
    
    if (inherits(result, "error") || is.null(result)) {
      chatgpt1 <- chat_openai(
        model = "gpt-5.1",
        system_prompt = id_prompt("")
      )
      fallback <- chatgpt1$chat(input_text)
      output_token_est <- estimate_tokens(fallback)
      total_output_tokens <<- total_output_tokens + output_token_est
      message("⚠️ Fallback preview:\n", substr(fallback, 1, 500))
      break
    }
    
    output_token_est <- estimate_tokens(paste(result$title %||% "", result$journal %||% "", result$authors %||% ""))
    total_input_tokens <<- total_input_tokens + input_token_est
    total_output_tokens <<- total_output_tokens + output_token_est
    total_requests <<- total_requests + 1
    
    message("✅ Success: ", basename(pdf_path))
    return(as_tibble_row(
      c(
        list(filename = basename(pdf_path), text = input_text, char_length = nchar(input_text), raw_json = list(result)),
        map(result, ~ if (is.null(.x)) NA_character_ else paste(as.character(unlist(.x)), collapse = "; "))
      )
    ))
  }
  
  message("❌ Final failure for ", basename(pdf_path), " after ", retries, " retries.")
  tibble(filename = basename(pdf_path), text = substr(full_text, 1, min_chars), char_length = min_chars)
}

extract_summary_single <- function(pdf_path, max_pages = 50, max_chars = 240000, max_retries = 6, global_pause = 15 * 60) {
  if (!file.exists(pdf_path)) {
    message("❌ File not found: ", pdf_path)
    return(NULL)
  }
  
  text_pages <- pdftools::pdf_text(pdf_path)
  truncated_text <- paste(text_pages[1:min(length(text_pages), max_pages)], collapse = "\n")
  truncated_text <- substr(truncated_text, 1, max_chars)
  estimated_tokens <- estimate_tokens(truncated_text)
  
  retries <- 0
  backoff <- c(60, 120, 180, 240, 300, 600)
  
  repeat {
    cat("📏 Estimated tokens:", estimated_tokens, "from", nchar(truncated_text), "chars\n")
    result <- tryCatch({
      print(Sys.time())
      gemini1 <- chat_google_gemini(
        model = "gemini-2.5-flash",
        system_prompt = id_prompt("")
      )

      gemini1$chat_structured(truncated_text, type = type_summary)
    }, error = function(e) {
      msg <- conditionMessage(e)
      if (grepl("429|quota|rate limit", msg)) {
        wait_time <- if (retries < length(backoff)) backoff[retries + 1] else 900
        wait_time <- runif(1, wait_time * 0.9, wait_time * 1.1)
        cat("⏳ Rate limit hit. Waiting", round(wait_time), "s (attempt", retries + 1, ")\n")
        Sys.sleep(wait_time)
        retries <<- retries + 1
        total_retries <<- total_retries + 1
        return("RETRY")
      } else {
        cat("❌ Unexpected error:", msg, "\n")
        return(NULL)
      }
    })
    
    if (identical(result, "RETRY")) {
      if (retries >= max_retries) {
        cat("❌ Max retries reached\n")
        return(NULL)
      }
    } else {
      break
    }
  }
  
  if (is.null(result)) {
    fallback <- gemini1$chat(truncated_text)
    cat("⚠️ Fallback preview:\n", substr(fallback, 1, 500), "\n")
    return(NULL)
  }
  
  result_list <- result
  result_list$filename <- basename(pdf_path)
  
  result_clean <- result_list %>% purrr::map_chr(~ {
    if (is.null(.x) || length(.x) == 0) return(NA_character_)
    if (is.atomic(.x) && length(.x) == 1) return(as.character(.x))
    if (is.atomic(.x) && length(.x) > 1) return(paste(.x, collapse = ", "))
    return(as.character(.x))
  })
  
  output_tokens <- estimate_tokens(paste(result_clean, collapse = " "))
  total_input_tokens <<- total_input_tokens + estimated_tokens
  total_output_tokens <<- total_output_tokens + output_tokens
  total_requests <<- total_requests + 1
  
  list(data = tibble::as_tibble_row(result_clean), token_count = estimated_tokens)
}
