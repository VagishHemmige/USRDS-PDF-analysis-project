Overview

This page explains the R code used to generate the transplant-focused results for the ATC analysis. The script starts from a wide-format dataset containing human review results and outputs from multiple large language models (LLMs), then produces summary tables and agreement visualizations.

The overall workflow has four main parts. First, the dataset is restricted to papers that were classified by human review as both new data analyses and transplant-related studies. Second, the script creates summary tables showing runtime and variable-level agreement between each LLM and the human reference abstraction. Third, it builds descriptive breakdown tables that show how often each model matched the human classification for each response category. Finally, it generates heatmaps to visualize agreement patterns across papers and variables.

Restricting the analytic dataset

The first step creates the analysis dataset used throughout the remainder of the script.

df_filt_atc <- df_wide %>%
  filter(
    is_this_paper_a_new_data_analysis_human == TRUE,
    is_this_a_transplant_related_study_human == TRUE

This code filters the full wide-format dataset to include only papers that the human reviewer classified as both:

  1. a new data analysis, and
  2. a transplant-related study.

This restriction is important because the downstream ATC results are intended to describe model performance within the subset of papers that were truly relevant to the transplant analysis. By applying the human-reviewed criteria first, the script ensures that agreement metrics are calculated on the final analytic sample rather than on the broader screening dataset.

Computing variable-level Cohen’s kappa

The next section computes Cohen’s kappa for each study variable and for each model, comparing model abstraction against the human reference standard.

Click to show/hide R Code 
kappa_results_atc <- purrr::map_dfr(human_vars, function(h) {
  base <- sub("_human$", "", h)

  tibble::tibble(
    variable = base,
    ChatGPT = compute_kappa(h, paste0(base, "_ChatGPT"), df_filt_atc),
    Claude  = compute_kappa(h, paste0(base, "_claude"),  df_filt_atc),
    Gemini  = compute_kappa(h, paste0(base, "_gemini"),  df_filt_atc)
  )%>%
    mutate(
    across(c(ChatGPT, Claude, Gemini), ~ ifelse(is.nan(.x), NA_real_, .x))
  )
})%>%
  filter(!variable %in% c(
    "which_pieces_of_data_were_abstracted_from_the_files",
    "which_tasks_would_not_be_covered",
    "which_type_of_transplant_study_is_this",
    "who_funded_the_study",
    "year",
    "journal",
    "title",
    "authors",
    "languages_used_SQL",
    "languages_used_Matlab",
    "languages_used_Julia"
  ))

This block loops over the vector human_vars, which contains the names of variables derived from human review. For each variable, the script removes the _human suffix to recover the shared base name, then uses that base name to identify the corresponding variables for ChatGPT, Claude, and Gemini.

The helper function compute_kappa() is then called three times per variable, once for each model. The result is a tidy table with one row per variable and one column per model.

After the kappas are computed, the script converts NaN values to missing values (NA). This is useful because some variables do not have enough variation, or may not have enough usable paired observations, to support estimation of Cohen’s kappa. Treating these as missing makes them easier to handle in later formatting steps.

Finally, a small group of variables is excluded from the kappa table. These are mostly free-text metadata fields or rarely used categories that were not intended to appear in the final agreement summary.

This section takes the raw kappa results and makes them easier to interpret. The helper function add_label_for_gt() adds a human-readable label for each variable, allowing the final table to display descriptive names instead of internal variable names.

Several formatting choices improve readability:

  • kappas are shown to three decimal places,
  • numeric columns are aligned on the decimal point,
  • missing kappas are displayed as - rather than NA,
  • related variables are grouped into sections such as programming languages, files used, and USRDS components used, and
  • indented rows visually distinguish subgroup items from the rest of the table.

The repeated footnotes explain why some kappa values are missing. This is an important interpretive detail, because missing kappa values usually reflect limited variation or insufficient paired observations rather than a coding error.

Finally, the call to data_color() applies the custom kappa_fill() function to color-code cells by agreement level. This makes it easier to scan the table and quickly identify variables with lower or higher agreement.

Building descriptive breakdown tables

In addition to kappa, the script generates a more descriptive table that shows the raw proportions of model responses within each human-coded category.

Click to show/hide R Code 
breakdown_table_atc <- map_dfr(human_vars, get_breakdown, df_filt = df_filt_atc)

This line applies the helper function get_breakdown() across all human-reviewed variables. Rather than summarizing agreement as a single kappa statistic, this step produces numerator/denominator counts and percentages for each model within each human category. These descriptive summaries complement the kappa table by showing the underlying distribution of matches.

The resulting dataset is then formatted as a gt table.

    label = md("**Gemini**"),
    columns = c(gem_nd, gem_pct)

    row_group.font.weight = "bold",

This table is designed to answer a different question from the kappa table. Instead of summarizing agreement with one statistic per variable, it shows the raw counts and percentages for each model. The final layout places each model under its own column spanner, with paired columns for:

  • numerator/denominator (Num/Den), and
  • percentage (%).

This is useful because a kappa value alone can sometimes hide important details about class imbalance or sparse categories. The breakdown table provides a more transparent view of how each model behaved for each response option.

Across all four heatmap sections, the helper function make_llm_agreement_heatmap() is doing the main work. Although its internal code is not shown here, its purpose is to transform variable-level agreement data into a paper-by-variable visual summary. This is useful because the tables describe average or category-level agreement, while the heatmaps reveal where disagreement clusters across individual papers.

Why both tables and heatmaps were used

The script intentionally uses both tabular summaries and graphical summaries because they answer different interpretive questions.

The kappa table provides a compact measure of agreement for each variable and each model. The breakdown table adds transparency by showing the underlying numerators, denominators, and percentages within each human-coded category. The heatmaps then add a paper-level view, showing where agreement and disagreement are concentrated across the dataset.

Together, these outputs provide a more complete picture of model performance than any single summary metric alone.

These saved objects make it possible to separate the computationally intensive analysis script from the reporting layer. In practice, the analysis can be run once, and the resulting images or .rds objects can then be loaded into Quarto pages without recomputing all tables and figures each time the site is rendered.

```{r eval=FALSE}
breakdown_gt_atc

saveRDS(breakdown_gt_atc, "ATC 2026 abstract/breakdown_gt_atc.rds")
```

This table is designed to answer a different question from the kappa table. Instead of summarizing agreement with one statistic per variable, it shows the raw counts and percentages for each model. The final layout places each model under its own column spanner, with paired columns for:

- numerator/denominator (`Num/Den`), and  
- percentage (`%`).

This is useful because a kappa value alone can sometimes hide important details about class imbalance or sparse categories. The breakdown table provides a more transparent view of how each model behaved for each response option.

# Why both tables and heatmaps were used

The script intentionally uses both tabular summaries and graphical summaries because they answer different interpretive questions.

The kappa table provides a compact measure of agreement for each variable and each model. The breakdown table adds transparency by showing the underlying numerators, denominators, and percentages within each human-coded category. The heatmaps then add a paper-level view, showing where agreement and disagreement are concentrated across the dataset.

Together, these outputs provide a more complete picture of model performance than any single summary metric alone.
